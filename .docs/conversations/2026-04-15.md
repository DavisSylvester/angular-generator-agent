### 2. Style Guide Extraction Phase (Box Model Decomposition)

**Request:** Add a pipeline step after the user picks a Stitch design that decomposes the design into atomic UI elements, generates a style guide, and uses it as the single source of truth for all component generation.

**Context:** The previous approach of matching entire screens in one Gemini call was producing inconsistent results — every fix introduced new regressions because there was no shared source of truth for individual design elements.

**New Approach: Box Model Decomposition + Style Guide First**

Instead of generating full pages and hoping they match:
1. **Step 1:** Decompose the Stitch design into atomic elements (side nav, header, buttons, cards, tables, badges, form fields, typography, spacing, color palette)
2. **Step 2:** Generate a style guide document (`.doc/style-guide.md`) — the single source of truth
3. **Step 3:** Generate components from the style guide (not screenshots)

**Files Created/Modified:**

| File | Change |
|------|--------|
| `src/types/design.mts` | Added `StyleGuide`, `StyleGuideElement`, `StyleGuideColor`, `TypographySpec`, `SpacingSpec` interfaces |
| `src/types/index.mts` | Exported new types |
| `src/orchestrator/style-guide-extraction.mts` | **New** — navigates to chosen Stitch design, screenshots it, sends to Gemini 2.5 Flash for box model decomposition, parses JSON response into structured `StyleGuide`, saves `.doc/style-guide.md` to workspace |
| `src/orchestrator/pipeline.mts` | Wired Phase 2a between design pick and save-decisions; passes `styleGuide` to component library agent and into enriched PRD for codegen |
| `src/agents/component-library-agent.mts` | Accepts optional `styleGuide` input; when present, uses precise pixel-level specs as source of truth instead of generic design notes |

**Pipeline Flow After Change:**

```
Phase 1:   Dribbble search → LLM design selection
Phase 2:   Stitch generation → User picks design
Phase 2a:  Style guide extraction (box model decomposition) ← NEW
Phase 2b:  Save decisions
Phase 3:   Component library (now uses style guide)
Phase 4:   Planning + codegen (enriched PRD includes style guide)
Phase 5:   Build validation + visual fidelity
```

### 3. Preflight Dependency Check with Auto-Install

**Request:** Create a step to ensure that the needed MCPs are installed; if not available, install them.

**Investigation:** Found that the project depends on:
- **Playwright MCP** (Claude Code plugin) — browser automation for all phases
- **Angular CLI** (`ng`) — scaffolding/building
- **Bun** — runtime
- **Node.js** — for Angular CLI
- **Google API Key** — for Gemini/Stitch

The pipeline uses a callback-based architecture where Playwright MCP tools are injected by the caller. In standalone mode, callbacks are no-ops.

**File Created:**

`src/orchestrator/preflight-deps.mts` — runs before Phase 0 and checks:

| Dependency | Check | Auto-install |
|---|---|---|
| Bun | `bun --version` | No (critical — aborts pipeline) |
| Node.js | `node --version` | No (critical — aborts pipeline) |
| Angular CLI | `ng --version` / `npx ng version` | Yes — `npm install -g @angular/cli` |
| Playwright MCP | Tests callbacks with `navigate(about:blank)` + `screenshot()` | Yes — installs `playwright` + `chromium` as fallback |
| Google API Key | Checks `GOOGLE_API_KEY` env var | No (non-critical warning) |

**Behavior:**
- Critical failures (bun, node) → pipeline aborts immediately
- Non-critical issues (Playwright stubs, missing API key) → warnings, pipeline degrades gracefully
- `PreflightReport` included in `PipelineResult` for visibility

**Files Modified:**

| File | Change |

| `src/orchestrator/preflight-deps.mts` | **New** — full preflight check module |
| `src/orchestrator/pipeline.mts` | Imported and wired preflight before Phase 0; added `preflightReport` to `PipelineResult`; early return on critical failure |
