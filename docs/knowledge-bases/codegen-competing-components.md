# Knowledge Base: Competing Component Overwrites

## Problem

When running multiple codegen batches sequentially, later batches can overwrite files generated by earlier batches. This is especially dangerous for:

1. **Root components** (`app.component.ts`) — a fix-pass for a child component (e.g., DataTable) may regenerate the root component as a demo harness instead of the router shell.
2. **Services** (`auth.service.ts`) — different batches may generate the same service with different method signatures, causing downstream consumers to break.
3. **Route-level components** — two batches may generate the same page (e.g., `LoginComponent`) under different paths (`auth/login/` vs `features/login/`), and the router lazy-loads the wrong one.

## Root Cause

The LLM is given a focused prompt per batch ("generate the DataTable component") but it often includes a demo `app.component.ts` to show usage. This overwrites the real root component. Similarly, when fixing build errors, the LLM may regenerate service files with simplified signatures that break existing consumers.

## Impact

- **Dashboard renders as plain text** instead of the rich Stitch-designed layout with metric cards, sport tabs, sessions, and activity feed.
- **Auth service loses methods** — `isLoggedIn`, `currentUser`, `getCurrentUser` disappear, causing build errors in guards and components that depend on them.
- **Login component duplication** — two versions exist under different paths with different implementations.

## Fix Applied

1. **Dashboard**: Regenerated with explicit Stitch design reference in the prompt — specifying exact colors (#0A192F, #0052CC, #F9FAFB), layout (metric cards row + sport tabs + sessions + activity feed), and Angular Material components (MatCard, MatIcon, MatTab, MatList).
2. **Auth service**: Manually restored with all required properties (isLoggedIn signal, currentUser signal, login(email, password), logout(), getCurrentUser(), isAuthenticated()).
3. **App component**: Replaced the DataTable demo with a simple `<router-outlet>` shell.

## Prevention (pipeline improvements needed)

1. **Lock critical files**: After generating `app.component.ts`, `app.config.ts`, `app.routes.ts`, and core services, mark them as locked — subsequent batches cannot overwrite them.
2. **File hash check**: Before writing a file, compare the hash of the new content with the existing content. If the file already exists and has meaningful content (not a stub), require explicit confirmation or skip the write.
3. **Service contract validation**: After each batch, verify that all services still expose the methods their consumers expect (grep for method calls and check they exist in the service).
4. **Visual fidelity check**: After codegen, compare built app screenshots against Stitch design screenshots. If the dashboard doesn't look like the design, trigger a targeted regeneration.

## Model Used for Fix

- **Model**: Gemini 2.5 Flash
- **Temperature**: 0.2
- **Prompt strategy**: Explicit Stitch design reference with hex colors, layout description, and required Angular Material components in the system prompt.

## Tokens Used

- Dashboard regeneration: ~2,000 input / ~3,500 output tokens
- Auth service: manual fix (no LLM tokens)
- App component: manual fix (no LLM tokens)
